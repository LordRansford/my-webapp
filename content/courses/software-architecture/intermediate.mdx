## Architecture styles in the real world

<SectionProgressToggle courseId="software-architecture" levelId="intermediate" sectionId="soft-arch-intermediate-architecture-styles" />

The big styles are patterns, not rules. A <GlossaryTip term="monolith">A single deployed system where most responsibilities live together.</GlossaryTip> can be right for speed. A <GlossaryTip term="modular monolith">One deployable system with clear internal modules.</GlossaryTip> gives structure without operational sprawl.

<GlossaryTip term="microservice">A small service that owns one clear job and deploys independently.</GlossaryTip> is great for team autonomy when you can manage the complexity. <GlossaryTip term="serverless">Functions that run on demand with managed infrastructure.</GlossaryTip> can be fast for spikes and small tasks.

<Callout variant="concept" type="info">
The best style is the one your team can operate safely, not the one that looks impressive on a slide.
</Callout>

## Integration and APIs

<SectionProgressToggle courseId="software-architecture" levelId="intermediate" sectionId="soft-arch-intermediate-integration-and-apis" />

Systems talk in two main ways. A <GlossaryTip term="synchronous call">A request where the caller waits for a response.</GlossaryTip> feels simple but creates tight coupling. An <GlossaryTip term="asynchronous message">A message sent without waiting for an immediate response.</GlossaryTip> decouples systems but adds coordination work.

Contracts matter. A <GlossaryTip term="contract">The agreed structure and behaviour of an API or message.</GlossaryTip> must be versioned so older clients keep working.

## Quality attributes and trade offs

<SectionProgressToggle courseId="software-architecture" levelId="intermediate" sectionId="soft-arch-intermediate-quality-attributes" />

<GlossaryTip term="latency">The time it takes for a request to receive a response.</GlossaryTip> and <GlossaryTip term="throughput">How many requests a system can handle in a period.</GlossaryTip> are easy to measure. <GlossaryTip term="availability">How often a system is able to serve requests.</GlossaryTip> is about reliability. We cannot maximise everything at once.

Architects balance these qualities rather than chase perfection. A fast system that fails often is not success.

## Data, state and consistency

<SectionProgressToggle courseId="software-architecture" levelId="intermediate" sectionId="soft-arch-intermediate-data-and-state" />

State hides everywhere. It lives in databases, caches, and message queues. <GlossaryTip term="consistency">How aligned different parts of the system are at the same time.</GlossaryTip> comes in two main flavours. <GlossaryTip term="eventual consistency">A style where parts may disagree for a short time, but they line up if we wait.</GlossaryTip> is common in distributed systems.

Distributed systems are hard because the network is slow and things fail. You decide where you accept delay and where you need certainty.
